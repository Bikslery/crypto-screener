# Password Reset via Telegram — Implementation Plan

## Goal

Add password recovery feature: user receives a 6-digit code via Telegram bot, enters it, then sets a new password. Two entry points:

1. **Login page (AuthModal)** — user must enter their login first
2. **Profile page (ProfileModal)** — login is already known (passed silently)

## Current Architecture

- **Auth**: JWT in httpOnly cookie, `bcryptjs` hash, Prisma `User` model
- **Telegram bot**: polling via `getUpdates`, `handleUpdate()` in `server/src/services/telegram/bot.ts`, `sendTelegramMessage()` helper
- **AuthModal**: step-based (`form` → `telegram` → `success`), tabs `login`/`register`
- **ProfileModal**: overlay modal, shows username + telegram status + logout
- **User model** (`schema.prisma`): `id`, `username`, `passwordHash`, `telegramChatId`, `telegramVerified`

## Proposed Approach

3-phase flow: **identify → verify code → set password**

### Prerequisite: Telegram must be bound

Password reset requires sending a code to the user's Telegram. If `telegramVerified === false`, the reset button should inform the user that Telegram must be bound first.

### Flow Details

**From AuthModal (login page):**
1. User clicks "забыли пароль?" link under the login form
2. Step changes to `reset-username`: shows a single input (login) + "отправить код" button
3. Server generates 6-digit code, stores it with expiry (5 min), sends via Telegram
4. Step changes to `reset-code`: shows 6-digit code input + countdown timer
5. User enters code → server validates
6. Step changes to `reset-password`: shows password + confirm password fields + "сменить пароль" button
7. Server updates password, user redirected to login form

**From ProfileModal (profile page):**
1. User clicks "сменить пароль" button in the "Действия" section
2. Inline sub-flow starts inside ProfileModal (no login input needed — userId from auth store)
3. Server sends code to user's bound Telegram
4. Code input → password input → success (same as above, but no username step)

## Step-by-Step Plan

### Step 1 — Prisma: Add `PasswordReset` model

**File**: `server/prisma/schema.prisma`

```prisma
model PasswordReset {
  id        String   @id @default(cuid())
  userId    String
  code      String   // 6-digit random code
  expiresAt DateTime // 5 minutes from creation
  used      Boolean  @default(false)
  user      User     @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@index([userId])
  @@index([code, userId])
}
```

Add relation to `User`:
```prisma
model User {
  // ... existing fields ...
  passwordResets PasswordReset[]
}
```

Then run `npx prisma migrate dev --name add-password-reset`.

### Step 2 — Server: Password reset routes

**File**: `server/src/routes/auth.ts` — add 3 new endpoints:

#### `POST /auth/reset-request`
- Body: `{ username?: string, userId?: string }` — one of them required
- Lookup user by username OR userId
- Validate: user must exist, `telegramVerified === true`, has `telegramChatId`
- Generate random 6-digit code (`Math.floor(100000 + Math.random() * 900000)`)
- Store in `PasswordReset` with `expiresAt = now + 5 min`
- Call `sendTelegramMessage(chatId, "Код для сброса пароля: 123456\nДействителен 5 минут.")`
- Return `{ ok: true, message: "Код отправлен в Telegram" }`
- Rate limit: max 1 request per minute per user (check last `PasswordReset` creation time)

#### `POST /auth/reset-verify`
- Body: `{ userId: string, code: string }`
- Find active reset: `userId` match, `code` match, `used === false`, `expiresAt > now`
- If not found → 400 error
- Mark reset as `used = true`
- Issue a short-lived **reset token** (JWT, 10 min expiry, different purpose — only for password change)
- Return `{ resetToken: "..." }`

#### `POST /auth/reset-password`
- Body: `{ resetToken: string, password: string }`
- Verify reset token (extract `userId`)
- Validate password (≥ 6 chars)
- Hash with bcrypt, update `User.passwordHash`
- Return `{ ok: true }`

### Step 3 — Server: Add reset token helpers

**File**: `server/src/middleware/auth.ts`

Add:
```ts
const RESET_JWT_SECRET = process.env.JWT_SECRET || 'change-me-in-production'

export function generateResetToken(userId: string): string {
  return jwt.sign({ userId, purpose: 'password-reset' }, RESET_JWT_SECRET, { expiresIn: '10m' })
}

export function verifyResetToken(token: string): string | null {
  try {
    const payload = jwt.verify(token, RESET_JWT_SECRET) as any
    if (payload.purpose !== 'password-reset') return null
    return payload.userId
  } catch {
    return null
  }
}
```

The existing `sendTelegramMessage()` is sufficient. The reset-request route calls it directly.

## Verification Steps

1. **Migration**: `npx prisma migrate dev` succeeds, `PasswordReset` table created
2. **AuthModal reset**: click "забыли пароль?" → enter login → code arrives in Telegram → enter code → set new password → login with new password works
3. **ProfileModal reset**: click "сменить пароль" → code arrives → enter code → set new password → login with new password works
4. **Edge cases**:
   - User without Telegram bound → error message "Сначала привяжите Telegram"
   - Expired code → error "Код истёк"
   - Wrong code → error "Неверный код"
   - Rate limit → error if requesting code more than once per minute
   - Non-existent username → error "Пользователь не найден"
5. **Security**: reset token expires in 10 min, one-time use code, old code invalidated

## Risks & Open Questions

1. **Telegram not bound** — user can't recover password. Consider email fallback in future. For now: show clear error.
2. **Rate limiting** — basic per-user check (last reset < 1 min ago). Could add Redis-based rate limiting later.
3. **Code entropy** — 6-digit = 1M combinations. 5-min expiry + rate limiting makes brute-force impractical.
4. **Multiple active resets** — old codes are invalidated when a new one is created (mark all previous as `used`).
5. **userId exposure** — `reset-request` returns `userId`. This is fine because `userId` alone doesn't grant access; the code + reset token are needed.
